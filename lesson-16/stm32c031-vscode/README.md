# Lesson 16 — Перше переривання (SysTick на STM32C031)

## Що нового

Уперше за весь курс **main() порожній** — він тільки налаштовує залізо і йде у нескінченний пустий цикл `while(1) { }`. Світлодіод моргає сам по собі завдяки **апаратному перериванню SysTick**.

Це фундаментальний перехід від _"послідовного скрипту з busy-wait"_ до _"подієво-орієнтованої системи"_. Так пишуть реальні embedded прошивки.

## Три нові концепції

### 1. SysTick — таймер у самому Cortex-M ядрі

**SysTick** — це 24-бітний down-counter який живе **усередині будь-якого Cortex-M процесора**. Не периферія STM32 — частина самого ядра. Тому він **однаковий** на STM32, TivaC, nRF52, RP2040 — будь-якому Cortex-M.

Має 4 регістри:

| Регістр | Призначення |
|---------|-------------|
| `SysTick->CTRL` | вмикання, вибір клоку, дозвіл переривання |
| `SysTick->LOAD` | значення з якого починається відлік (24-біт) |
| `SysTick->VAL`  | поточне значення лічильника |
| `SysTick->CALIB`| калібрувальна інформація (зазвичай не використовується) |

Логіка: лічильник зменшується від `LOAD` до `0`, при досягненні `0` — **генерує переривання** і починає відлік знову з `LOAD`.

```
LOAD = 5_999_999    ;  при SYSCLK=12 MHz → 6 млн циклів = 0.5 с
                         ↓ ↓ ↓ ↓
                       5999999, 5999998, ..., 1, 0, IRQ!, 5999999, ...
```

### 2. Перекриття `SysTick_Handler`

У стартап-коді (`startup_stm32c031xx.c`) `SysTick_Handler` оголошений **weak**:

```c
void SysTick_Handler (void) __attribute__ ((weak, alias("Default_Handler")));
```

Це означає _"за замовчуванням викликати Default_Handler, але якщо хтось напише свій SysTick_Handler — використати його"_. Тому в `bsp.c` достатньо просто визначити функцію з ТАКИМ САМИМ ІМ'ЯМ:

```c
void SysTick_Handler(void) {
    GPIOA->ODR ^= (1U << LD4_PIN);
}
```

Linker побачить що тепер є **сильний** символ `SysTick_Handler` — і використає його у vector table замість weak alias. Жодних змін у startup не треба.

### 3. `__enable_irq()` — глобальний дозвіл переривань

Після reset процесор стартує з **PRIMASK=1** — це апаратний "стоп-кран": усі maskable переривання заблоковані. Поки ти не дозволиш — SysTick може дзвонити скільки завгодно, ядро не реагуватиме.

```c
__enable_irq();    /* виконує інструкцію CPSIE i — clear PRIMASK */
```

З цього моменту переривання починають обслуговуватися.

## Чому `BSRR` тут не годиться, а `ODR` — годиться

У lesson-04..13 ми писали в `BSRR` — атомарний регістр **set/reset**. Тут нам треба **toggle** (`XOR`), а `BSRR` toggle не підтримує — там 16 бітів "set", 16 "reset", без "flip".

Toggle потребує:
1. Прочитати поточний стан
2. Інвертувати потрібний біт
3. Записати назад

Це **read-modify-write** на регістрі `ODR` (Output Data Register), який R/W і завжди відображає поточний стан виходів:

```c
GPIOA->ODR ^= (1U << LD4_PIN);   /* read → XOR → write */
```

⚠️ **Race condition**: якщо два переривання одночасно роблять `ODR ^= ...` на двох різних бітах — можна загубити одну зміну. На однопоточному baremetal (нічого не перериває SysTick) це безпечно. У RTOS — треба захист.

## Чому LD4 моргає рівно раз на секунду

```c
SysTick->LOAD = SYS_CLOCK_HZ / 2U - 1U;
                12_000_000 / 2 - 1 = 5_999_999
```

→ Переривання кожні **0.5 секунди**. Кожне перемикає LED. Один повний цикл (ON-OFF) = **1 Hz**.

Якщо хочеш блимання 2 Hz — постав `LOAD = SYS_CLOCK_HZ / 4 - 1`. Якщо 0.5 Hz — `LOAD = SYS_CLOCK_HZ - 1`.

## Що подивитись у дебагері

1. **F5** → потрапиш у `main()`. Натисни **Continue**.
2. Через ~0.5 с — LD4 загориться. Ще через 0.5 — згасне. І так далі.
3. **Постав breakpoint** у `SysTick_Handler` (на `GPIOA->ODR ^= ...`).
4. Continue — зупиниться у handler'і **на кожному перериванні**.
5. У **Call Stack** побачиш як CPU зайшов: `main` → ⚡ → `SysTick_Handler`.
   Реально CPU не "викликав" handler — це залізо стрибнуло за адресою з vector table.
6. У **CORTEX REGISTERS** подивись `PRIMASK` (має бути `0` після `__enable_irq`).
   Подивись `xPSR` — у полі `ISR_NUMBER` побачиш `15` (SysTick exception). У main() це поле = `0`.

Це і є **жива анатомія переривання**.
